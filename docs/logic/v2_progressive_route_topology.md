# 人生路况渐进式路线拓扑 v2

日期：2026-05-30

## 1. 产品判断

人生路况不是一次性给结论。用户从起点出发，在每个节点面对 1 到 2 个选择。系统按出生信息、节气、干支、五行气和当下规则给出路线建议，但交互上让用户逐步选择。

这里的拓扑是“展示骨架”，不是命理结论本身。

拓扑定义的是：

- 地图上有哪些位置。
- 每个位置可以承载哪类选择。
- 路线如何分叉、合流、抵达终局。

规则层计算的是：

- 当前用户处在哪个阶段。
- 当前节点该出现哪些选择。
- 每个节点使用哪个 icon。
- 每个节点的具体意义、适合、不宜和解释。
- 最终更接近哪一种人生可能性。

因此，拓扑可以调整，但必须版本化。不能每次生成时任意变形，否则地图、节点坐标、icon 映射和用户回溯都会失去稳定性。

核心体验：

1. 先看到当前位置。
2. 再看到眼前两条路。
3. 选一条路后，下一段只展开相关可能性。
4. 每一步都说明处境、适合、不宜和代价。
5. 最后收敛为 3 到 5 种人生走向。

第一版先用 4 个终局，降低交互复杂度。

## 1.1 拓扑的可调整边界

### 固定部分

- 起点在左下。
- 人生方向从左下向右上延伸。
- 路线只往前，不回头。
- 每个交互节点最多给 1 到 2 个选择。
- 每屏主展示不超过当前相关节点，非当前路线置灰。
- 终局收敛到 3 到 5 种人生可能性。

### 可调整部分

- 第一层可以是 2 条或 3 条起步路线。
- 中间节点数量可以从 2 层扩展到 3 层。
- 分支可以合流。
- 终局可以从 4 个扩展到 5 个。
- 每个节点的 icon、文案和建议由规则层计算，不写死。

### 不应调整的部分

- 不应把拓扑画成随机散点。
- 不应把所有节点一次性全亮。
- 不应把 icon 固定为某个节点的永久含义。
- 不应让 LLM 自行决定路线结构。
- 不应让视觉地图脱离规则层坐标。

## 2. 渐进式披露规则

### 2.1 默认态

- 只显示当前位置和第一组选择。
- 不展示完整结果。
- 不一次性铺满所有 icon。
- 地图可以看到未来路形，但未选择路线上的节点置灰。
- 每一步最多展示 1 到 2 个可点击选项。

### 2.2 选择后

- 已选路线高亮。
- 未选路线保留为灰态，表示“另一种可能”。
- 下一节点展开新的 1 到 2 个选择。
- 底部卡片先给短概述，详情向上展开。

### 2.3 终局

- 用户完成 2 到 3 次选择后，系统给出终局。
- 终局不是判命，是“这条走法更可能形成的生活状态”。
- 终局必须可回溯到用户每一步选择。

## 3. 四层拓扑

第一版采用四层结构：

```text
L0 起点
  ├─ L1-A 先稳住
  │   ├─ L2-A1 继续推进
  │   │   ├─ L3-A1a 扩大责任 ─┐
  │   │   └─ L3-A1b 保持节奏 ─┤
  │   └─ L2-A2 补足条件       ├─ O1 稳步成局
  │       ├─ L3-A2a 先补资源 ─┤
  │       └─ L3-A2b 先等信号 ─┘
  │
  └─ L1-B 试机会
      ├─ L2-B1 小步外探
      │   ├─ L3-B1a 借力打开 ─┐
      │   └─ L3-B1b 轻量试错 ─┤
      └─ L2-B2 避开消耗       ├─ O2 窗口打开
          ├─ L3-B2a 绕开冲突 ─┤
          └─ L3-B2b 放慢观察 ─┘

交汇规则：
L3-A1b + L3-B2b -> O3 平衡转向
L3-A2b + L3-B2a -> O4 延后发力
```

## 4. 节点定义

节点只定义“槽位”，不定义固定人生意义。下面的“用户看到”和“内部含义”是默认示例，用于设计底图和第一版交互；正式结果里，意义必须由命盘、节气、干支、五行气、规则分数和用户选择路径共同生成。

| 层级 | 节点 | 用户看到 | 内部含义 | 可用 icon |
|---|---|---|---|---|
| L0 | START | 你现在站在起点 | 当前人生阶段与当下处境 | today_general / direction_choice |
| L1-A | A | 先稳住 | 稳定、主线、承重 | stability_anchor / keep_lane |
| L1-B | B | 试机会 | 机会、外探、窗口 | opportunity_window / lane_change |
| L2-A1 | A1 | 继续推进 | 主线已有推力 | momentum_boost / career_path |
| L2-A2 | A2 | 补足条件 | 根基不足，先补资源 | earth_base / safe_zone |
| L2-B1 | B1 | 小步外探 | 有外部回音，但不可押满 | support_lighthouse / relationship_contact |
| L2-B2 | B2 | 避开消耗 | 风险、冲突、情绪成本偏高 | risk_barrier / detour_route |
| L3-A1a | A1a | 扩大责任 | 可承接更大任务 | career_path / money_commitment |
| L3-A1b | A1b | 保持节奏 | 稳中推进，不急变道 | keep_lane / slow_down |
| L3-A2a | A2a | 先补资源 | 先补钱、时间、关系或能力 | wealth_safe_chest / support_lighthouse |
| L3-A2b | A2b | 先等信号 | 条件未明，等待明确回音 | wait_signal / near_exit |
| L3-B1a | B1a | 借力打开 | 通过人、平台或机会进入 | support_lighthouse / opportunity_window |
| L3-B1b | B1b | 轻量试错 | 低成本尝试，快速拿反馈 | lane_change / accelerate_now |
| L3-B2a | B2a | 绕开冲突 | 不正面硬碰，换路通过 | detour_route / conflict_marker |
| L3-B2b | B2b | 放慢观察 | 先保状态，再看合流点 | mood_rest / slow_down |

## 4.1 节点运行字段

每个节点进入系统时，至少需要以下字段：

```json
{
  "node_id": "L2-A1",
  "topology_role": "choice",
  "stage_range": "near_future",
  "available_choices": ["L3-A1a", "L3-A1b"],
  "computed_signal": "momentum_with_stability",
  "selected_icon": "momentum_boost",
  "meaning_source": "bazi_rule_engine",
  "copy_source": "llm_rewrite_under_standard",
  "is_active": true,
  "is_clickable": true
}
```

字段说明：

- `node_id`：地图坐标和路径关系。
- `topology_role`：起点、选择点、风险点、合流点、终局。
- `stage_range`：当下、近未来、中段、远段、终局。
- `available_choices`：下一步可走路线。
- `computed_signal`：规则层算出的信号，不直接展示给用户。
- `selected_icon`：由信号映射出的 icon。
- `meaning_source`：意义来源必须是规则层，不是静态节点文案。
- `copy_source`：文案可由 LLM 优化，但不能改逻辑。
- `is_active`：当前路径是否高亮。
- `is_clickable`：当前是否允许用户点击。

## 5. 终局定义

| 终局 | 用户看到 | 说明 | 适合 | 不宜 |
|---|---|---|---|---|
| O1 | 稳步成局 | 主线能承重，结果来自持续推进。 | 把确定的事做深。 | 频繁换方向。 |
| O2 | 窗口打开 | 外部机会出现，但要小步验证。 | 递出请求，测试回音。 | 一次押满。 |
| O3 | 平衡转向 | 原路和新路都有价值，需要保留接口。 | 边走边校准。 | 情绪化切断旧路。 |
| O4 | 延后发力 | 当前不是没路，而是火候未满。 | 补资源、等信号。 | 为了焦虑硬推。 |

## 6. 文案标准

### 6.1 起点文案

标题：

```text
你现在面临两种走法
```

说明：

```text
未来的人生路漫长。先看眼前这一步：一条更稳，一条更开。
```

### 6.2 选择卡模板

```text
{路线名}
{一句处境}
适合：{一件直接动作}
不宜：{一个明确避开的动作}
```

### 6.3 直接措辞词库

优先使用：

- 适度放慢脚步
- 多做积累
- 先把基础打稳
- 先看回音
- 小步试一下
- 暂时不要押满
- 先避开消耗
- 保留回旋余地
- 把确定的事做完
- 等信号更清楚

少用或不用：

- 守主脉
- 绕暗流
- 补根基
- 火候未满
- 入局
- 起势
- 顺局
- 承气

命理词可以放在展开详情里，但路线名和按钮必须用白话。

## 7. 给模型的输出约束

模型必须输出当前节点文案，而不是一次性输出全局结论。

输出字段：

```json
{
  "current_node": {
    "id": "START",
    "headline": "你现在面临两种走法",
    "summary": "一条更稳，一条更开。先选眼前这一步。",
    "stage_hint": "当下"
  },
  "choices": [
    {
      "id": "A",
      "label": "先稳住",
      "situation": "适合先把基础打稳。",
      "suitable": "把确定的事做完。",
      "avoid": "不要急着换方向。"
    },
    {
      "id": "B",
      "label": "试机会",
      "situation": "有窗口，但要小步试。",
      "suitable": "先看一次回音。",
      "avoid": "不要一次押满。"
    }
  ],
  "locked_hint": "后面的路会随你的选择展开。"
}
```

## 8. 底图道路 Mapping

前端分两层：

- 映射层：`topology_node_mapping.json` 识别底图上的道路圆盘和连接线，形成 `road_anchors` 与 `road_paths`。
- 逻辑层：命理计算只决定当前选择哪条路、展开哪些节点；节点必须绑定到 `road_anchors`，不能自由散点。

当前规则：

- 起点固定在 `START`。
- 第一层只有两条可选路线：`L1-A 先稳住`、`L1-B 试机会`。
- 选择 A 后，只展开 A 路上的 `L2-A* / L3-A* / O*`。
- 选择 B 后，只展开 B 路上的 `L2-B* / L3-B* / O*`。
- 共享节点只能用于合流或共同判断，不得让 A 的未来节点显示在 B 的道路上。

设计判断：

- 路是底层逻辑。
- 节点是路线上的选择点。
- Icon 是节点上的命理信号。
- 文案解释用户为什么在这个节点看到这个信号。
