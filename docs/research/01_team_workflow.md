# Team Workflow

## Objective

Build a human-reviewed research system before product design.

The app must feel friendly and supportive. The logic must be traceable.

## Roles

### Peter

Role: 算命大师.

Owns:
- 术数体系 selection.
- Rule interpretation.
- Term accuracy.
- Master-level response standard.

Output:
- Accepted rule systems.
- Explanation rules.
- Red lines for wrong or shallow claims.

### Andrew

Role: data expert.

Owns:
- Public data search.
- Open-source project scan.
- Dataset quality review.
- License and maintenance check.

Output:
- Source inventory.
- Engine candidates.
- Data gaps.
- License risks.

### Bill

Role: product manager.

Owns:
- User needs.
- Product flow.
- MVP scope.
- Product requirements.
- Human review gates.

Output:
- Product spec.
- Feature list.
- Content policy.
- Evaluation criteria.

## Review Gates

### Gate 1: System Selection

Decision:
- Which 玄学 systems enter MVP.
- Which systems stay research-only.

Approval owner: user.

### Gate 2: Source Approval

Decision:
- Which libraries and datasets can be used.
- Which sources require manual validation.

Approval owner: user.

### Gate 3: Product Spec Approval

Decision:
- MVP scope.
- User journey.
- Output style.
- Safety boundaries.

Approval owner: user.

## Current Recommendation

Start narrow.

MVP system:
- Daily 运势: calendar, 干支, 五行, 黄历宜忌.
- Personal layer: 八字基础盘.
- Optional later: 紫微斗数, 周易问答.

Reason:
- Daily product frequency is clear.
- Calculation can be audited.
- Output can be personalized.
- Risk is lower than broad fortune claims.
